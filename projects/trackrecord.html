Why it exists

Understanding how public discussion moves around an event — a product recall, a policy announcement, a campaign — usually means stitching together scrapers, notebooks, and dashboards by hand. TrackRecord makes that a single command: it collects public discussion from many source types, normalizes everything into one schema, runs a local analysis stack, and synthesizes the results into executive summaries, monitoring reports, and alerts.

It's also an exercise in trustworthy demo engineering: the entire system runs against committed synthetic fixtures with real accuracy/F1 evaluation numbers, so anyone can verify the pipeline end to end before pointing it at live sources.

Distinctions the analysis layer refuses to blur

Tone is not stance

VADER measures how a text sounds; the stance module scores the author's position toward the target. "This recall is blown out of proportion — Aurora handled it responsibly" is negative in tone but supports Aurora. TrackRecord computes, stores, and reports both separately — a distinction most sentiment dashboards silently collapse.

Topics are not narratives

Topics answer "what is discussed." Narratives follow each topic week over week and label it emerging, fading, burst, or steady — turning a static snapshot into a story with direction.

  • Real evaluation, not vibes. webintel eval reports accuracy/F1 of the analysis modules against committed ground truth; demo runs write the numbers to eval.json.
  • Pluggable connectors. Subclass BaseConnector, decorate with @register, normalize into Item, drop a fixture — the core never changes.
  • Polite by default when live. Rate-limited to 30 req/min with exponential backoff, robots.txt-aware, custom User-Agent; run manifests account for every request before you scale up.
  • Full CLI surface: run, ingest, analyze, eval, search (FTS5), schedule for recurring monitoring, and deterministic fixture regeneration.

Engineering notes

Offline fixtures and live fetching share the same parsing code — the demo isn't a mock of the system, it is the system pointed at committed data. Live mode is a documented, user-initiated step, keyless where possible.

The optional LLM planner path improves planning on vague goals, but the deterministic local planner handles everything without it — an availability-first posture unusual in agent frameworks.
