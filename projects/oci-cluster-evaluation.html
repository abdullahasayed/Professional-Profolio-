OCI-Cluster-Evaluation

Why it exists

SemiAnalysis' ratings of GPU cloud offerings have become a de facto industry gold standard — strong enough that heavyweights like Oracle pay attention. Much of the criteria behind ClusterMAX is published plainly, yet actually running those tests on your own cluster still demands senior-engineer intuition scattered across a dozen tools.

This project turns that criteria into a playbook. The goal: even a novice cloud engineer can get robust, industry-recognizable feedback on their cluster's particulars — while senior engineers get a fast suite that highlights weaknesses in metrics the whole industry recognizes.

What's in the toolkit

Five focused scripts plus a Make interface, each with a --help screen:

  • Baseline audit — captures what the cluster actually exposed at run time: GPU visibility, drivers, environment. Text artifacts, reviewable later.
  • WAN / package smoke — times package installs and internet egress (uv, pip, curl), proxy- and mirror-aware, with full download manifests and checksums.
  • Storage smoke — sequential and random behavior via fio, with tunable size and runtime for quick confidence checks vs. heavier passes.
  • Nsight Compute smoke — validates non-root profiler access against a minimal CUDA vector-add, a notorious silent failure on managed clusters.
  • SLURM NCCL all-reduce template — a multi-node collective-communications smoke with an overridable nccl-tests binary path.
# the ladder, in practice make test # laptop-safe repo checks make baseline BASELINE_OUTDIR=artifacts/baseline make wan-smoke WAN_OUTDIR=artifacts/wan-run1 make fio-smoke TARGET_DIR=/data SIZE=4G RUNTIME=30 make ncu-smoke sbatch examples/slurm/nccl_allreduce.sbatch # 16-GPU collective smoke

Engineering notes

The design bias is evidence over output: a failed smoke produces one directory you can hand to another engineer — config, versions, timings, checksums — rather than a scroll of terminal text. That's what makes runs comparable and proxy/mirror debugging tractable.

Commands are explicitly tiered by blast radius, from "safe on any laptop" to "requires a working multi-node SLURM environment" — the README literally answers "which command should I run?" This grew out of real consulting: assessing OCI cluster performance and scheduling behavior under SLURM and Kubernetes, where a repeatable diagnostics runbook saved the client from external specialists.
